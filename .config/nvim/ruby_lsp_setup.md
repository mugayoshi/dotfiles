# Ruby LSP Setup (anews)

## Overview

**ruby_lsp** is the only active LSP server for Ruby files. It handles completions, diagnostics, formatting, and go-to-definition for both gems and project classes.

---

## Files

| File | Purpose |
|------|---------|
| `~/.config/nvim/lsp/ruby_lsp.lua` | ruby_lsp server config |
| `~/.config/nvim/lua/core/lsp.lua` | enables the server via `vim.lsp.enable` |
| `<project>/.ruby-lsp/Gemfile` | custom bundle for ruby_lsp (auto-generated, do not commit) |
| `<project>/.ruby-lsp/Gemfile.lock` | pinned versions — must stay at ruby-lsp 0.26.x + ruby-lsp-rails 0.4.x |

---

## Key config (`ruby_lsp.lua`)

```lua
cmd_env = {
  RUBY_LSP_BYPASS_TYPECHECKER = '1',
},
```

This project includes `sorbet-static` in its Gemfile. Without this env var, ruby_lsp detects Sorbet and stops handling go-to-definition for project-defined constants (expecting a Sorbet LSP to handle them instead). Setting `RUBY_LSP_BYPASS_TYPECHECKER=1` prevents that detection, so `gd` works on project classes like `BoxAPIToken`.

---

## .ruby-lsp bundle

The `.ruby-lsp/Gemfile` is auto-generated by ruby_lsp with loose constraints (`ruby-lsp >= 0.18.0`). The **Gemfile.lock must stay pinned to 0.26.x** — ruby-lsp 0.23.x has a Ruby 3.4 incompatibility (`no implicit conversion of Pathname into String` during indexing) that breaks go-to-definition entirely.

If the lock file gets regenerated and reverts to 0.23.x:

```sh
BUNDLE_GEMFILE=.ruby-lsp/Gemfile bundle update ruby-lsp ruby-lsp-rails
```

---

## Known limitations

- `grr` (references) — not supported by ruby_lsp
- `K` (hover) — returns nothing for project classes; ruby_lsp defers hover to Sorbet even with `BYPASS_TYPECHECKER` set
